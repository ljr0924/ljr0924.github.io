Python 设计模式：工厂方法模式

定义

工厂方法模式是一种创建型设计模式。它定义了一个创建对象的接口，但将对象的实际创建推迟到子类中进行。

工厂方法模式通过让子类决定实例化哪一个具体类，实现了将对象的创建和使用分离。

这样，在不改变工厂类接口的情况下，通过创建不同的工厂子类就可以创建不同类型的对象。

使用场景

  • 对象创建过程的变化

当对象的创建过程可能因为不同的条件（如不同的用户角色、不同的业务逻辑分支等）而发生变化时，工厂方法模式很有用。

例如，在一个电商系统中，根据用户是普通用户还是管理员用户，创建订单对象的过程可能有所不同，此时可以使用工厂方法模式，让不同的工厂子类负责创建不同类型用户对应的订单对象。

  • 需要符合开闭原则的场景

开闭原则要求软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。

如果系统需要频繁地添加新的产品对象类型，并且希望在不修改现有代码的情况下完成扩展，工厂方法模式是一个很好的选择。

例如，一个图形绘制系统，可能会不断添加新的图形类型，使用工厂方法模式，通过创建新的工厂子类来创建新图形，而不用修改原有的工厂类和其他使用图形对象的代码。

代码结构

  • 抽象产品（Abstract Product）

定义了产品对象的抽象类或接口，它规定了产品对象的公共方法和属性。

所有具体的产品类都要实现这个接口或者继承这个抽象类

  • 具体产品（Concrete Product）

抽象产品的具体实现类，它们实现了抽象产品中定义的方法，提供了具体的功能

  • 抽象工厂（Abstract Factory）

包含一个抽象的工厂方法，用于创建产品对象。这个方法的返回类型是抽象产品类型。

  • 具体工厂（Concrete Factory）

继承自抽象工厂类，实现了抽象工厂方法，用于创建具体的产品对象。

优点

  • 符合开闭原则

当需要添加新的产品对象时，只需要创建新的具体工厂类和具体产品类，而不需要修改现有的工厂类和产品类代码。

例如在交通工具制造系统中，如果要添加一种新的交通工具，如摩托车，只需要创建摩托车工厂类和摩托车产品类，原有的汽车工厂和自行车工厂等代码不受影响。

  • 解耦对象创建和使用

客户端代码只需要依赖抽象工厂和抽象产品类，不需要了解具体产品的创建细节。

这样使得代码的职责更加清晰，降低了代码的耦合度。

例如，在使用交通工具的代码中，只需要通过工厂类获取交通工具对象并使用，不需要关心交通工具是如何制造出来的。

缺点

  • 工厂子类过多

每增加一种新的产品，就需要创建一个新的工厂子类，这可能会导致工厂子类过多，使代码变得复杂。

例如，如果有很多种交通工具需要创建，就会有很多个对应的工厂子类，增加了代码的管理和维护成本。

  • 代码复杂度增加

相比于简单的对象创建方式，工厂方法模式引入了抽象工厂、抽象产品、具体工厂和具体产品等多个层次的类，这使得代码结构更加复杂，对于初学者或者简单的项目来说，可能会增加理解和实现的难度。
